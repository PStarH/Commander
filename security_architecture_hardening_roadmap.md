## 二、为什么 100% 安全不可能

### 根本限制

1. **停机问题**: 无法枚举所有可能的程序输入和状态组合。总存在未被预见的攻击路径。

2. **LLM 概率性**: 大语言模型的输出不是确定性的。攻击者可以通过语义变换(同义词、隐喻、多语言混合)绕过任何基于模式匹配的注入检测。

3. **零日漏洞**: 依赖中的未知漏洞永远存在。XZ Utils 后门(CVE-2024-3094)在开源代码中隐藏了 3 年,通过了所有自动化安全扫描。

4. **侧信道**: 时序攻击、功耗分析、电磁辐射——这些物理通道绕过所有逻辑控制。

5. **供应链根攻击**: 编译器后门(Ken Thompson 的 "Reflections on Trusting Trust")、硬件固件后门——应用层无法检测。

6. **社会工程**: 最坚固的架构也防不住拿到合法凭证的攻击者。

### 正确的目标设定

**不是**: "防住所有攻击"(不可能)
**而是**: "让攻击成本远超攻击收益"

衡量指标:
- 攻击者需要多少资源(时间、计算力、专业知识)才能突破
- 单层突破后,下一层是否仍然有效(纵深)
- 检测到攻击后,系统是否能自动响应(闭环)

## 三、可立即实现的加固 (实现缺口修复)

以下 4 项缺口可以通过纯代码修改立即修复,无需架构重设计:

### G4: 记忆投毒检测接入运行时

**当前**: `memoryPoisoningDetector` 模块完整存在,但仅作为 HTTP API 按需调用。Agent 运行时写入 ThreeLayerMemory 时完全不经过此检测器。

**修复**: 在 `agentRuntime.ts` 的记忆写入路径中,强制调用 `detectPoisoning()`,检测结果为 `block` 时拒绝写入。

**影响**: 防止攻击者通过工具输出污染记忆,RAG 检索时激活(研究显示 >80% 成功率 @ <0.1% 污染)。

### G5: 密钥保险库实际采用

**当前**: `encryptedSecretsVault.ts` 设计完善(AES-256-GCM, HKDF, 密钥轮换),但 LLM API 密钥仍直接从 `process.env` 读取。

**修复**: 将 provider key 加载改为从 vault 解密读取,环境变量仅存储 vault 主密钥。

### G7: Egress 过滤强化

**当前**: `NoopSB` 在无 OS 级沙箱时作为兜底,完全无网络过滤。`network: 'proxy'` 模式可被原始 socket 绕过。

**修复**: 生产环境禁用 `NoopSB`——无可用沙箱时拒绝执行。`proxy` 模式叠加 iptables/网络命名空间强制。

### G8: CPU 资源限制

**当前**: Docker 沙箱有 `--memory` 但无 `--cpus`。恶意 agent 可消耗全部 CPU。

**修复**: 添加 `--cpus` 和 `--cpu-quota` 参数。

## 四、需要架构设计的加固 (中期路线图)

### G1: RASP 实时阻断引擎 (最关键的架构缺口)

**问题**: 所有安全检测器(注入检测、DLP、幻觉检测、记忆投毒)都是"开环"的——检测到攻击后仅发告警到 MessageBus,无任何自动阻断动作。

**设计**: 实现 `SecurityResponseEngine`,订阅 `security.alert` topic,执行响应动作:
- 暂停 agent 会话
- 撤销 capability token
- 隔离 agent 到只读模式
- 切断 agent 网络访问
- 触发安全快照(取证)

**技术可行性**: 可纯 Node.js 实现,不需要内核级支持。

### G2: 信息流控制 (IFC) / 污点追踪

**问题**: 无数据标签系统。无法区分"来自外部网页的数据"和"来自可信系统的数据",因此无法阻止间接提示注入。

**设计**: 
- 为所有数据流添加 taint 标签(`trusted` / `untrusted` / `external`)
- 工具输出默认标记为 `untrusted`
- `untrusted` 数据不得流入: 系统提示词、工具调用参数(出站工具)、记忆写入
- 标签随数据流动自动传播(字符串拼接时取最严格标签)

**技术可行性**: 需要封装 LLM 消息构建、工具调用、记忆写入三条路径。工作量较大但可行。

### G3: A2A mTLS 双向认证

**问题**: A2A 通信仅用 Bearer token,无传输层双向认证。Token 泄露即被冒充。

**设计**:
- 引入内部 CA (Certificate Authority)
- 每个 agent 实例生成客户端证书
- A2A 连接要求客户端证书验证
- 证书与 agent 身份绑定(包含 agentId 扩展字段)

**技术可行性**: Node.js `tls` 模块原生支持 mTLS。需要证书生成和轮换基础设施。

### G6: 沙箱逃逸运行时检测

**问题**: 仅日志记录,无运行时 syscall 监控。

**设计**:
- Linux: 使用 eBPF 程序监控容器内异常 syscall(mount, nsenter, /proc/1/root)
- macOS: 使用 Endpoint Security framework
- 检测到逃逸行为时立即终止容器并告警

**技术可行性**: 需要 eBPF 程序开发(C/Rust),超出应用层范围。可考虑集成 Falco 或 Tracee。

### G10: Agent 间记忆隔离

**问题**: 同一租户内多 agent 共享 memory store,无 per-agent 命名空间强制隔离。

**设计**: 
- 记忆条目增加 `writerAgentId` 字段
- 读取时按 agent 可见性策略过滤
- 引入 agent 间数据共享策略(默认隔离,显式授权共享)

## 七、残余风险声明

即使完成上述全部修复,以下风险仍然存在:

1. **LLM 模型本身的漏洞**: 模型权重中的后门、训练数据投毒——应用层无法检测
2. **零日漏洞**: 依赖中的未知漏洞——需要持续的依赖更新和监控
3. **内核级攻击**: 容器逃逸到宿主机内核——需要硬件级隔离(如 confidential computing)
4. **社会工程**: 合法凭证被窃取——需要 MFA 和行为分析
5. **物理侧信道**: 时序、功耗分析——需要硬件级防护

**安全是一个持续过程,不是一次性目标。** 架构应该设计为可演进的安全态势管理,而非静态的防御工事。
