# Commander Service Level Objectives

This document describes the public Service Level Objectives (SLOs) for Commander.
The targets are measured daily by the SLO benchmark runner
(`.github/workflows/slo-bench.yml`) and reported in the web dashboard at `/slo`.

## SLO Summary

| SLO                            | Target     | Measurement Window | Evidence                              |
| ------------------------------ | ---------- | ------------------ | ------------------------------------- |
| Run submission API availability | ≥ 99.95%  | 30d rolling        | `apps/api` health-check metrics       |
| P95 schedule latency            | ≤ 5 s     | 24h rolling        | `sloMonitoringEngine` latency samples |
| Worker failure step recovery   | ≤ 60 s     | per incident       | `reclaimExpiredLeases` + timer logs   |
| DLQ recovery success rate      | ≥ 99.5%   | 24h rolling        | `deadLetterQueue` replay metrics      |
| Event-log hash-chain integrity | 100%       | per write          | `eventSourcingEngine` verification    |
| Tool approval fail-closed rate | 100%       | per request        | `approvalSystem` deny logs            |

## RPO/RTO Targets

| Metric | Target |
|--------|--------|
| RPO    | ≤ 5 minutes |
| RTO (in-region) | ≤ 60 minutes |

These targets are validated by the quarterly disaster recovery drill
(`scripts/dr-backup-verify.ts`) and documented in `docs/runbooks/dr-backup-restore.md`.

## Status

The latest measured baseline is stored in `docs/baselines/slo-baseline.2026-07-08.json`.
It is generated by `scripts/bench-slo-baseline.ts` and validated on every push
by `pnpm test tests/e2e/sloMeasurement.test.ts`.

The SLO definitions in `packages/core/src/observability/sloOperations.ts`
(`DEFAULT_SLO_CONFIG`) are the authoritative source; this document must stay
in sync with them.

## Dashboard

The web UI exposes these objectives without authentication at `/slo`. The page
reads the committed baseline JSON and renders the current status for operators
and customers.
