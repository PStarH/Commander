# TELOS Framework Design

> **TELOS** (τέλος) — "The Ultimate Purpose"
> 代号：打造不可被超越的究极多 Agent 效率系统

---

## 一、愿景

TELOS 不是"又一个多 Agent 框架"。

它是：
- **人类团队管理千年智慧的代码化**：从古希腊议会到现代 OKR，从甘特图到 Scrum，从拜占庭容错到洛克菲勒纪律
- **多 Agent 系统所有 SOTA 研究的集大成**：Anthropic 的治理模型、MemPalace 的记忆架构、LangGraph 的状态机、微软的编排模式、Google 的 A2A、Meta 的协调策略
- **在每一条维度上碾压所有竞争者**：Token 消耗、延迟、吞吐量、治理正确率、经验保留率、人机协作效率

**目标：不是"做得好"，是"没有对手"**。

---

## 二、核心理念：两个无限

### 2.1 无限借鉴人类经验

人类团队解决问题已经积累了数千年经验：
- **决策**：从苏格拉底问答法到现代决策树
- **协调**：从军队命令链到 Spotify Squad Model
- **知识管理**：从索引卡片到 Roam Research
- **风险管理**：从拜占庭将军问题到现代精算
- **效率优化**：从泰勒科学管理到敏捷迭代

TELOS 把这些提炼成**可执行的代码约束**，不是提示词，是规则。

### 2.2 无限榨取系统潜力

每个多 Agent 系统都有浪费的三个地方：
- **Token 浪费**：重复上下文、冗余记忆、粗糙的预算
- **时间浪费**：同步阻塞、串行执行、无意义的轮询
- **经验浪费**：每次失败都是独立的，历史经验全部丢失

TELOS 的每个组件都设计为：
> "不浪费一个 Token，不浪费一次计算，不浪费一条经验"

## 三、TELOS 的六大护城河

### 3.1 自适应编排引擎（Adaptive Orchestration Engine）

**不是"选择一种模式"，是"为每个任务定制最优模式"**。

输入：任务描述 + 风险 + 复杂度 + 上下文
输出：为这个任务精确设计的执行计划

5 种基础模式（可组合）：
- `SEQUENTIAL`：简单任务，单 Agent，Token 最省
- `PARALLEL`：独立子任务，最大并发，最快完成
- `HANDOFF`：专家链式传递，每步给最合适的人
- `MAGNETIC`：开放式探索，允许多路径同时试错
- `CONSENSUS`：高风险决策，多模型投票，BFT 容错

超越所有框架的关键：TELOS 同时考虑 **任务特征 + Agent 状态 + 上下文预算 + 历史成功率** 来选模式，而不是只看任务复杂度。

### 3.2 三层记忆体系（Tri-Layer Memory Architecture）

受 MemPalace 96.6% LongMemEval 启发的完整记忆系统：

```
┌─────────────────────────────────────────┐
│ L0: Working Memory (无限快)            │
│     当前上下文，50 条上限，LRU 淘汰     │
│     → 每次 Agent 调用都从这里读        │
└─────────────────────────────────────────┘
                    ↓ 晋升
┌─────────────────────────────────────────┐
│ L1: Episodic Memory (经验层)           │
│     完整情景记忆，带时间衰减 + 重要性  │
│     → 反思引擎写入，相似任务检索       │
└─────────────────────────────────────────┘
                    ↓ 固化
┌─────────────────────────────────────────┐
│ L2: Semantic Memory (知识层)           │
│     结构化经验，模式 + 规则 + 决策     │
│     → 策划阶段引用，指导计划生成       │
└─────────────────────────────────────────┘
                    ↓ 归档
┌─────────────────────────────────────────┐
│ L3: Procedural Memory (程序层)         │
│     如何做事的 SOP，工具使用，流程     │
│     → 执行阶段自动调用，不需要重新学   │
└─────────────────────────────────────────┘
```

**比 MemPalace 更强**：MemPalace 只有"存储和查找"，TELOS 有完整的 **晋升/衰减/检索/遗忘** 机制。

### 3.3 宪法级治理（Constitutional Governance）

三层治理模式：

**AUTO（自主执行）**：
- 低风险任务全自动
- 所有 Agent 平等执行
- 只记录日志，不干预

**GUARDED（守望模式）**：
- 中等风险任务，Sentinel 全程监控
- 高风险操作自动降级为 PROPOSE
- 实时生成风险评估报告

**MANUAL（审批模式）**：
- 高风险任务，强制人类/Commander 审批
- 审批必须带理由，写入 DECISION 记忆
- 未审批的操作不可执行

**护城河**：治理规则不是提示词，是**硬编码约束**。MANUAL 模式下，API 直接返回 409，不存在绕过路径。

### 3.4 经验进化引擎（Experience Evolution Engine）

从 Reflexion 框架升级的四层反思：

**Evolution（成功后提炼）**：
- 什么做对了？
- 可以抽象成什么规则？
- 下次如何更快/更省？

**Fix（失败后根因分析）**：
- 为什么失败？
- 是哪一步的决策错误？
- 需要修改什么规则/流程？

**Rejection（边界情况标记）**：
- 这个任务超出当前能力范围
- 需要人类介入还是继续尝试？

**Contradiction Detection（自我纠错）**：
- 新经验是否与旧经验矛盾？
- 矛盾时保留哪个？如何记录冲突？

### 3.5 共识与决策层（Consensus & Decision Layer）

BFT（拜占庭容错）共识 + 人类决策的最优混合：

```typescript
if (任务风险 >= CRITICAL) {
  // 强制多模型共识
  投票结果 = await BFTVote(agents, proposal, 3);
  if (投票结果.confidence < 0.8) {
    // 共识不足，降级为人类决策
    return await 人类审批(proposal);
  }
}
```

**护城河**：大多数框架的"共识"只是多数投票。TELOS 有完整的 BFT 容错、置信度评估、升级决策机制。

### 3.6 持续进化架构（Continuous Evolution Architecture）

每个任务完成后自动运行：

```
任务完成 → 反思 → 经验入库 → 策略更新 → 下次更快
                ↓
           效果验证
                ↓
         策略是否改进？
           ↙        ↘
        是           否（回滚）
        ↓              ↓
    固化策略       记录教训
```

**护城河**：其他框架的经验是"存储"，TELOS 的经验是"活的"，会自动影响下一次决策。

## 七、TELOS 的不可复制性

任何竞争者想追上 TELOS，需要同时：
1. 再造 Phase 1 的 7 个核心组件
2. 实现完整的 4 层记忆 + 晋升机制
3. 写一套宪法级治理 + 审批流
4. 建立进化引擎 + 自我纠错
5. 积累足够多的真实任务数据来训练策略模型
6. 打通所有观测层和审计层

**这不是 6 个月能完成的，是 2-3 年的持续迭代**。

而 TELOS 在他们追赶的过程中，会因为：
- 更多的任务数据 → 更好的策略
- 更多的经验 → 更低的 token 消耗
- 更多的失败案例 → 更强的纠错能力

**持续拉开差距**。
