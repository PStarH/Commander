## 3. 逐条核对

### 3.1 §2 隔离级别与默认

**裁决**：GAP

**证据**：
- `manager.ts:101-110` `getProfile()` 只读 `COMMANDER_SANDBOX_MODE`，**完全没有 `COMMANDER_SANDBOX_ISOLATION`** 解析。
- `manager.ts:74-99` `getSandbox(mechanism)` 直接返回 `this.sandboxes[0]`，不区分 docker/gvisor 选型。
- `platforms.ts:697` 注释明确 "Falls back to DockerSB if runsc is not available"，与 spec §2 "gvisor 不得降级" 直接冲突。
- `platforms.ts:1248-1277` `discoverSandboxes()` 把 seatbelt/bwrap/AppContainer/docker/gvisor/TEE 全部纳入候选，按平台探测返回首个可用，**无 `process` 级别 Backend 命名**。
- `discoverSandboxes()` 仅在 `NODE_ENV=production` 且无任何候选时 throw，缺 `process` 级别在 production 拒启的硬闸。

**缺失**：
- `COMMANDER_SANDBOX_ISOLATION` 配置接口与解析函数
- `process` Backend 命名空间（区分 host exec 与 bwrap/seatbelt/seccomp）
- gvisor 不可用时的显式 `throw SandboxInitializationError`（替代静默 fallback）

### 3.2 §3 生产禁止项

#### 3.2.1 构建期门禁

**裁决**：GAP（CRITICAL）

**证据**：
- `manager.ts:21-24` `allowNoSandboxFallback()` 直接读 `process.env.COMMANDER_ALLOW_NO_SANDBOX`，无任何构建期替换。
- `manager.ts:32-35` `allowUncheckedExecution()` 同源问题。
- `pluginSandbox.ts:87-94` `resolveMode()` 直接读 `process.env.COMMANDER_PLUGIN_SANDBOX`。
- 全仓搜索 `COMMANDER_ALLOW_NO_SANDBOX` 仅 2 处命中（manager.ts + spec），**无构建脚本扫常量**。
- 全仓搜索 `esbuild.*define|webpack.*DefinePlugin|swc.*define` 在 build 配置中无命中。

#### 3.2.2 运行时禁止项

**裁决**：GAP（CRITICAL）

**证据**：
- `manager.ts:51-57` `if (this.sandboxes.length === 0 && !this.allowNoSandbox) throw` —— 当 `COMMANDER_ALLOW_NO_SANDBOX=true` 时**不抛**，与 spec §3 "无沙箱旁路 拒绝条件 = `COMMANDER_ALLOW_NO_SANDBOX=1/true/yes`" 冲突。
- `manager.ts:85-91` `getSandbox()` 在 `allowNoSandbox=true` 时显式 `return new NoopSB()`，与 spec §4.1 "不得切换到 Noop" 冲突。
- `pluginSandbox.ts:286-306` `COMMANDER_PLUGIN_SANDBOX_SOFT=1` 时软回退 in-process；spec §3 明确禁止 `COMMANDER_PLUGIN_SANDBOX_SOFT` truthy。
- `pluginSandbox.ts:193-201` `required` + `production` 才硬拒 in-process；显式 `COMMANDER_PLUGIN_SANDBOX=in_process` + 非 production 可绕过。
- `localBackend.ts:100-146` `rejectOnNoSandbox` 是构造函数参数，**不是环境变量也不是 policy 强制**；默认无沙箱时回退到 `execFileSync`（line 124）。

### 3.3 §4.1 Worker 启动顺序

**裁决**：GAP（CRITICAL）

**证据**：`packages/worker-plane/src/bootstrap.ts:40-50`：

```typescript
export async function createWorkerService(): Promise<WorkerService> {
  // ── Validate required env vars ──
  const dbUrl = process.env.DATABASE_URL;
  if (!dbUrl) {
    throw new Error('DATABASE_URL is required for worker bootstrap');
  }

  const authToken = process.env.COMMANDER_WORKER_AUTH_TOKEN;
  if (!authToken) {
    throw new Error('COMMANDER_WORKER_AUTH_TOKEN is required for worker bootstrap');
  }
```

- 只校验 `DATABASE_URL` 和 `COMMANDER_WORKER_AUTH_TOKEN`。
- **未构造 `SandboxManager`**。
- **未读取 `COMMANDER_SANDBOX_ISOLATION`**。
- **未触发沙箱能力探针**。
- **未抛 `SandboxInitializationError`**。
- 整个文件 244 行中没有任何对 `sandbox/` 子目录的引用。

`atr/workerPool.ts:38-99` `KernelWorkerPool.start()` 直接进入 claim 循环，不做沙箱预检。

直接违反 spec §4.1 五步启动顺序与安全不变量。

### 3.4 §4.2 工作负载启动失败 fail-closed

**裁决**：PARTIAL

**证据**：
- `manager.ts:172-173` `getSandbox()` 失败会 throw `SandboxInitializationError`，确实不会执行命令。
- `manager.ts:127-168` 沙箱逃逸检测器加载/执行失败会 `blockedByDetector()` 拒绝。
- `atr/workerPool.ts:119-131` `dispatch` 异常路径仅 `inFlight--` 和 `reportSilentFailure`，**不主动释放 lease**，lease 状态转换归属 `scheduler.ts`（未审计）。

**缺口**：
- tenant/run/step 维度的 lease 释放协议在 sandbox 失败路径上未显式声明。
- `platforms.ts:1268-1274` `discoverSandboxes()` 直接 `throw new Error(...)` 而非 `SandboxInitializationError`，异常类型不统一。

### 3.5 §5 租户/工作负载策略

**裁决**：GAP（HIGH）

**证据**：
- 全仓 `tenantSandboxPolicy|TenantSandboxPolicy|tenantSandboxPolicy|sandboxPolicy` 搜索无命中。
- `sandbox/` 目录下无 `tenantId`/`workloadId` 字段，类型定义（`types.ts`）仅含 `SandboxMechanism`、`SandboxProfile`、`SandboxExecutionResult`。
- `DockerSB.execute()` 使用 `-v ${workdir}:${workdir}` 挂载调用方提供的工作目录，**没有 `commander-sbx-<opaque-workload-id>` 容器命名**。
- `DockerSB.execute()` 镜像 allowlist 5 个 base image（node:22-slim, node:20-slim, python:3.12-slim, python:3.11-slim, ubuntu:22.04），**未与租户/工作负载维度关联**。

**缺失**：
- `TenantSandboxPolicy` 数据结构与持久化（YAML/SQLite）
- 容器名服务端生成函数（拒绝用户输入容器名）
- 每 workload 临时 volume / 网络 namespace
- 镜像 digest 锁定（不仅是 tag allowlist）

### 3.6 §5.2 强制策略

**裁决**：PARTIAL

**证据**：
- `platforms.ts:752-759` GVisorSB 有 `--cap-drop ALL`、`--security-opt no-new-privileges`、`--read-only`、`--tmpfs /tmp:rw,noexec,nosuid,size=64m`。
- `platforms.ts:1023-1031` DockerSB 同样有上述四项。
- `DockerSB` **未设置 `--user`/`-u` 非 root**；`isValidContainerName` 只验证容器名格式（dockerExecBackend.ts:7-11）。
- `platforms.ts:38-52` `filterEnv()` 有 deny+allow 双重过滤，`EXTRA_DENY` 覆盖 16 项 secret 模式，但 `HARDENED` profile 的 `envVarAllowList` 仅 13 项（profiles.ts:133-147），与 spec §5.2 "allowlist；默认剔除 token、secret、password、credential、数据库连接串和 Docker/SSH 控制变量" 匹配。
- `platforms.ts:46` 已剥离 `DOCKER_*` / `SSH_*` 前缀变量。

**缺口**：
- Docker 容器未强制非 root user。
- tenant scope 在 Worker claim / sandbox 创建 / 执行审计三处一致性校验未实现。

## 5. 审计结论

WS7 spec 的设计目标（生产 fail-closed、构建期封死旁路、租户维度隔离、CI 守门）与现有实现存在**结构性差距**：

1. **核心安全不变量（生产 Worker 完成沙箱能力校验前不得 claim/execute）当前未达成**——突破口在 `bootstrap.ts`。
2. **5 项生产禁止项中有 4 项在当前代码可达**（无沙箱旁路、未检查执行旁路、插件 in_process、gvisor 降级）。
3. **构建期门禁完全缺失**；CI 反向打开旁路。
4. spec 自我陈述的 4 个 WS7 缺口与本次审计完全一致，且范围比 spec 列出的更大：缺租户策略存储、缺容器名规范、缺每 workload 卷/网络 namespace。

**审计建议**：

本 spec 不应进入 Phase 2 Build。需先在 Phase 1 评审中形成正式决议后再行实现：

1. `COMMANDER_SANDBOX_ISOLATION` 的解析与选型函数由 sandbox 包持有；gvisor 选型失败时显式 `throw SandboxInitializationError` 而非静默 fallback。
2. `worker-plane/bootstrap.ts` 在 `createWorkerService()` 入口处强制调用一次 SandboxManager 能力探针，并把拒绝错误原样向上抛。
3. `.github/workflows/ci.yml` 移除全部 `COMMANDER_ALLOW_NO_SANDBOX: true`（可改为只在 dev/test job 局部启用），新增 `ws7-boot-refuse.yml` 独立 workflow：生产配置 + 探针失败 → 进程非零退出 + 无任何副作用。
4. 租户/工作负载策略层（tenantId/runId/stepId/workloadId 注入点）独立成 `TenantSandboxPolicy` 模块，存储与 `profiles.ts` 平级；每 workload 容器名按 `commander-sbx-<opaque-workload-id>` 服务端生成，禁止用户传入。
