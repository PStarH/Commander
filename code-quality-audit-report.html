Top 10 优先修复项 P0

  1. 拆分 God Object — agentRuntime.ts (4832行, execute()~3192行) 和 orchestrator.ts (execute()~900行) 是全项目最大的可维护性风险，任何修改都有高回归风险。按阶段拆分为独立类。
  2. 修复健康检查未接线 — httpServer.ts 三处 new HealthCollector() 未传入真实数据源，熔断器/DLQ/补偿/Provider 五项检查恒返回 "healthy — not wired"，对运维具有欺骗性。
  3. 修复并发数据竞争 — agentRuntime.ts 中 this.governor/this.slidingWindow 等实例字段在 execute() 内被按运行重赋值，并发 execute() 会互相覆盖，导致租户隔离失效。
  4. 修复 teeEnclave 命令注入 — teeEnclave.ts:622 生成含 eval 的 shell 脚本，环境变量值未经转义直接 eval，存在命令注入风险。
  5. 移除 postQuantumCrypto 伪装实现 — postQuantumCrypto.ts:355 的 generateSharedSecret 是 HMAC 占位符却伪装成后量子密钥协商，调用方可能误以为已获得 PQ 安全保证。
  6. 修复 Datadog 导出器 — datadogExporter.ts:127 含拼写错误 header DD-EVIL-AGENT，且使用 logs intake endpoint 发送 span 数据，Datadog APM 无法识别。
  7. 修复 Provider 模型选择反转 — 5 个 provider (openai/deepseek/glm/mimo/xiaomi) 用 this.defaultModel || request.model，路由器选定的模型被静默丢弃。
  8. 修复 A2A Server 无认证 — a2aServer.ts:163 的 message/send 端点可触发 runtime.execute() 执行任意 goal，完全无认证。
  9. 添加 Web Error Boundary — apps/web 无任何 Error Boundary，任何渲染异常会白屏崩溃整个应用。
  10. 清理 harness 死代码与重复 — BaseHarness 完全死代码、4 个 harness 重复 emitEvent/steerQueue、McpHarness 空桩、harnessRegistry -10 优先级不可达。

核心执行引擎，承担LLM路由、Agent循环执行、工具编排、可靠性治理（熔断/DLQ/补偿）、健康检查、HTTP服务与多租户隔离。150+文件。

agentRuntime.ts:1159-4350

C1postQuantumCrypto.ts — generateSharedSecret 是伪装成后量子密钥协商的占位符
postQuantumCrypto.ts:355-367
方法名为"使用PQ安全密钥协商生成共享密钥"，注释声称"Ready for ML-KEM-768"，但实际实现是HMAC-SHA512(randomSalt, peerKey + privateKey)。这不是非对称密钥协商，不提供任何后量子安全性。

H2sandboxVerifier.ts — resource_limits 验证为全零占位符
sandboxVerifier.ts:378
resource_limits检查项为{pass:0, fail:0, skip:0, error:0}。沙箱验证器无法实际验证内存/CPU限制是否被强制执行。全零可能被误读为"通过"。

rotationSignoffVerifier.ts:439, 773, 934

Sandbox 模块 (3.0/5)

C1teeEnclave.ts — buildNitroVsockProxy 生成含 eval 的 shell 脚本，存在命令注入
teeEnclave.ts:622, 631
第622行eval "export $(echo "$ENV" | tr '\n' ' ')" — $ENV来自环境变量值，若包含shell元字符（如; rm -rf /），eval会执行注入命令。
将环境变量逐个export，对值进行单引号转义，避免eval。或改用env命令前缀传递环境变量。
H1teeEnclave.ts — execSyncSafe 及清理逻辑使用字符串插值执行命令
teeEnclave.ts:652-653, 294, 299, 311, 667
execSyncSafe接受任意字符串命令直接execSync(cmd)。当前调用点输入安全，但"字符串插值→execSync"模式极其脆弱。
改为execFileSync(binary, args[])形式，强制调用方以参数数组传递。
H2appContainer.ts — 使用 S-1-15-2-1 (ALL_APP_PACKAGES) 授权，权限过宽
appContainer.ts:185
icacls /grant "*S-1-15-2-1":... 会授予所有AppContainer配置文件访问权限，而非仅当前创建的容器。违背最小权限原则。
使用Get-AppContainerProfile获取特定容器的SID进行授权。

亮点

  • capabilityToken.ts 密码学实现优秀 — HMAC-SHA256签名 + timingSafeEqual防时序攻击 + crypto.randomBytes + 委托深度限制 + canonical JSON。
  • auditChainLedger.ts 防篡改哈希链 — HKDF-SHA256派生每租户密钥、per-process chainId、seq gap/broken_link/invalid_hmac多维篡改检测。
  • encryptedSecretsVault.ts — AES-256-GCM + 每版本独立随机salt+IV + HKDF域分离 + 版本轮换 + 解密审计日志。
  • localBackend.ts fail-closed — 无沙箱时检查SHELL_UNSAFE_RE拒绝shell元字符，使用execFileSync（无shell）执行。
  • sshBackend.ts 输入验证 — isValidShellPath防路径穿越、isValidSshHost验证主机名、StrictHostKeyChecking=yes防中间人。
  • dockerExecBackend.ts — 容器名作为独立spawn参数传递（非字符串插值），从根本上消除注入。

总结与建议

Commander 是一个架构雄心极大的多智能体编排系统，融合了 ROMA、DOVA、FoA、Saga、Thompson Sampling 等多种学术与工程模式。项目的类型纪律整体优秀（全仓 : any 仅 8 处），安全核心模块（capabilityToken、auditChainLedger、encryptedSecretsVault）密码学实现规范，ATR 事务运行时的分布式模式实现达到教科书级。

但项目存在三个系统性风险：

1. God Object 阻碍演进

agentRuntime.ts（4832行）和 orchestrator.ts（execute ~900行）是全项目最大的技术债务。它们使任何修改都有高回归风险，且几乎无法单元测试。建议优先按阶段拆分，利用已有的 phases/ 目录和 baseOrchestrator.ts 抽象。

2. 占位符伪装成实现

多个安全关键模块（postQuantumCrypto、adaptiveHitl、sandboxVerifier）存在占位符返回看似有效实则无功能的结果，对调用方具有欺骗性。建议统一标注 @unimplemented 并在调用处抛出错误，而非返回默认值。

3. 重复与散落增加维护成本

补偿系统三重重复、记忆能力三重散落、harness 四重重复、provider 11/23 重复造轮子。建议建立清晰的分层契约，以公共接口/基类收口。

本报告基于对 300+ 源文件的逐模块深度审查，所有问题均附具体文件路径与行号。审查覆盖 packages/core 全部 25+ 子模块、packages/ 下 6 个独立包、apps/ 下 2 个应用。
